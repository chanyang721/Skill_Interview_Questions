### 자바스크립트의 장점과 단점에 대해서 알려주세요
#### 자바스크립트의 장점
* 자바스크립트는 인터프리터 언어이기 때문에 실행 속도가 컴파일 언어에 빠르다는 장점과, 배우는데 비교적 쉬운 구조를 가지고 있습니다.
* 거의 모든 브라우저가 지원하기 때문에 많은 개발자들이 다양한 애플리케이션을 만들때 사용하는 언어로 사용되고 있습니다. 
* npm(node package manageer)을 통해 다양한 모듈을 사용할 수 있습니다.
* 백엔드에서도 사용 가능한 nodeJS라는 런타임 환경을 가지고 있어 서버 개발에도 사용할 수 있습니다.
* 다른 프로그래밍 언어와 호환되는 관계를 가지고 있기 때문에 임베드가 가능합니다.
#### 자바스크립트의 단점
* 자바스크립트 코드는 사용자가 손쉽게 확인 할 수 있기 때문에, 클라이언트 보안에 취약하며
* 다양한 브라우저에서 지원되지만 각 브라우저가 자바스크립트를 컴파일하는 인터프리터가 다르기 때문에 배포 전에 모든 브라우저에서 잘 돌아가는지 확인해야 합니다. 
* 코드상에 에러가 하나라도 있으면 렌더링이 전부 중지되기 때문에 소스 코드에 대한 테스트를 반드시 해야합니다.
* Java나 TypeScript의 class에서 제공하는 은닉화 키워드인 private을 지원하는 브라우저가 거의 없다는 단점이 있습니다.
* 추상화 기능 부제 => 인터페이스를 설정할 방법이 없다. 

### Node.js가 무엇인가요 ? 
* Nods.js는 V8엔진과 libuv를 이용하여 웹 서버에서 JavaScript를 비동기적으로 사용할 수 있도록 해주는 런타임 환경(프로그램)입니다.
* 특히 node.js는 libuv 라이브러리로 Non-blocking I/O와 단일 스레드 방식의 이벤트 루프를 사용하여 높은 처리 성능을 가지고 있는게 특징입니다.

### Node.js가 싱글스레드인 이유와 장점을 설명해주세요 (중요) 
* nodejs가 싱글스레드인 이유는 자원에 대한 동시성을 신경쓰지 않기 위해 싱글 스레드로 만들었습니다. 그리고 nodejs의 장점으로는 자바스크립트를 알고 있다면 새로운 언어를 익히지 않고 서버를 만들수 있다는 것과, 싱글 스레드를 기반으로 백그라운드에서 멀티 스레드의 이벤트를 처리하는 방식을 사용하여 비동기를 구현했기 때문에 서버에 무리를 주지 않습니다. 또한 npm을 통해 다양한 모듈을 제공받아 사용할 수 있다는 것이 있습니다.
* 단점이 있다면, 싱글 스레드이기 때문에 하나의 작업 자체가 오래 걸리는 서비스에는 어울리지 않다는 것과, 인터프리터 언어이기 때문에 에러가 나느 경우 프로세스 자체가 꺼지기 때문에 반드시 모든 경우에 대한 소스코드의 테스트를 병행하면서 개발을 해야 한다는 것이 있습니다.

### Node가 싱글 스레드인데 멀티 스레드보다 빠른 이유를 설명해주세요.
* 싱글 스레드는 동기화 작업을 하지 않아도 되기 때문에 멀티 스레드보다 빠릅니다.
* 멀티스레드를 사용하는 경우 공유 자원으로 인한 문제 해결을 위해 동기화에 신경을 써야합니다. 하지만, 싱글 스레드는 이 동기화에 신경쓰지 않아도 되기 때문에 멀티 스레드보다 빠르다고 알고 있습니다. 

### Node.js를 사용하는 이유 ? 
* 싱글스레드 기반의 비동기 I/O처리로 매우 빠른 고성능 서버를 만들 수 있으며, 많은 사용자로 인해 만들어진 다양한 모듈을 npm을 통해 사용할 수 있다는 장점이 있기 떄문에 사용합니다.
* 스크립트 언어의 특성 상 한줄 씩 읽어내려가며 실행되기 때문에 만약 하나의 에러라도 발생하게 된다면, 서버 자체가 다운되는 단점이 있어 반드시 소스 코드에 대한 테스트 과정을 거쳐야합니다.
* 개발은 빠르고 쉽다는 장점이 있으나, 운영 관점에서 테스트, 장애 대응, 디버깅 등이 약점이다.

### array vs linkedlist 차이점
* 요소의 접근 방식에서의 차이점은 배열의 경우 인덱스를 알고 있다면, O(1)의 시간 복잡도를 가지게 되며, 링크드 리스트의 경우 처음부터 순서대로 접근하여 찾아야 하기 때문에 O(N)의 시간복잡도를 가지게 됩니다.
* 요소의 저장 방식에서의 차이점은 배열은 인접한 메모리 위치에 저장되지만, 링크드리스트의 경우 다음 요소와만 연결되어 저장됩니다.
* 요소의 삽입과 삭제에서의 차이점은 배열의 경우 인덱스로 접근하는 O(1)의 시간 복잡도를 갖지면 접근한 요소를 삭제하는 경우 인덱스 값을 모두 변경해야하는 O(N)의 과정을 거쳐야 합니다. 하지만, 링크드리스트의 경우 요소들을 자신 다음의 요소값의 정보만 저장하는 특징이 있기 때문에 이 저장된 값만 변경하면 되어 O(1)의 시간복잡도를 가지게 되지만, 이 요소에 접근하기 위해서 처음부터 찾아야 하는 O(N)의 시간 복잡도를 가지게 됩니다.
* 결론적으로 이미 저장된 데이터에 접근하는 것이 주 용도인 경우 배열을 사용하고, 데이터를 수정, 삭제가 주 업무인 경우에는 링크드리스트를 사용하는 것이 시간복잡도 측면에서 이점을 얻을 수 있습니다.

### Hash Table이란? 
* (Key, Value)로 데이터를 저장하는 자료구조로서, O(1)의 시간 복잡도를 가진 검색속도를 제공하기 위해 사용됩니다. 내부적으로는 배열(버킷)을 생성하여 index값으로 실제 데이터를 저장합니다. 해시테이블에는 버킷의 인덱스 값을 계산하여 버킷에 저장하게 됩니다. 인덱스를 구하기 위해 함수를 한번만 실행하면 되기 때문에 O(1)의 시간복잡도를 가집니다.
* 만약 해시 함수가 중복된 인덱스로 충돌되게 된다면, O(N)의 시간 복잡도를 가진 분리 연결법과 개방 주소법을 사용한다고 알고 있습니다. 
### HashMap(해시맵)과 HashTable(해시테이블) 차이
* 두 방식의 차이는 동기화 지원 여부에 있으며, 동기화를 고려해야 하는 경우 해시 테이블을 사용하고, 동기화가 필요 없다면 해시맵을 사용합니다.

### jwt 토큰을 사용하신 이유는 뭔가요. 어떻게 구성되어 있나요.
* jwt 토큰을 사용하는 이유는 stateful한 상태를 유지시키기 위한 유저의 인증 정보를 암호화 하기 위해 사용됩니다.
jwt 토큰을 사용하는 이유는 유저의 정보를 암호화하여 클라이언트와 서버간에 stateful한 상태를 유지시키기 위해 사용합니다.
* stateless한 서버의 특성상 요청으로 받은 state값을 따로 저장하지 않기 떄문에 로그인 인증과 같이  jwt의 구성으로는 header, payload, 그리고 signature의 3부분으로 나누어져 있으며, header에는 토큰 종류, 암호화 알고리즘의 종류가 저장되어 있고, payload에는 유저 정보, 권한과 같은 정보를 넣게 되며, 시그니쳐 부분에는 헤더와 페이로드를 인코딩한 값과 솔트를 적용한 암호화된 값이 포함되어 있습니다.

### 세션과 토큰 중의 각각 장단점을 말씀해주세요.
* 세션은 유저 인증 방식으로 유저의 정보를 서버의 세션스토리지에 저장하고 클라이언트에게는 세션아이디만 전달하여 세션아이디를 통해 유저를 인증하는 방식입니다.
* 그렇기 때문에 세션의 장점과 단점은 서버에 유저 정보를 저장한다는 특징에서 말할 수 있다고 생각합니다. 우선 장점으로는 서버에서 유저 정보가 있기 때문에 추가적인 확인이 가능하다는 것이구요. 단점으로는 서버의 저장 공간을 차지하기 서버에 무리를 줄 수 있습니다. 또한, 세션아이디를 쿠키를 사용하여 전달하기 때문에 XSS공격을 당할 가능성이 있습니다.
* 토큰도 유저 인증을 위한 방법이지만, 세션과는 다르게 서버에서 암호화된 토큰을 클라이언트에 저장하여 해당 토큰의 유효성을 통해 유저 인증하는 방법을 사용하고 있습니다. 
* 토큰의 장점으로 암호화된 토큰만을 사용하여 유저 인증을 진행하기 떄문에, 인증을 위한 저장 공간이 없다는 것과, 이미 암호화된 토큰을 주고 받기 때문에 암호화 키를 노출시키지 않는다.
* 하지만, 단점으로는 토큰은 로그아웃 기능이 없기 떄문에 만료시간으로 컨트롤 해야한다는 것이 있습니다.

* dependencies에는 프로그램 동작에 반드시 필요한 모듈들을 명시해놓고, devDependencies에는 프로그램 동작에 영향을 주지 않지만 필요한 모듈들을 넣습니다. 
* srcipts에는 CLI 환경에서의 명령어를 작성하여 "npm + 명령어"를 입력할 시 해당 명령어의 value로 작성한 문자열을 실행됩니다.

### Get,Post 메소드 차이점 (중요)
* Get과 Post의 차이점은 HTTP 요청을 통해 데이터베이스의 값의 변경 유무라고 할 수 있습니다. 
* Get 메소드는 HTTP 요청 시 단순히 요청에 의한 값을 받아기 위해 사용되지만, Post 메소드는 이와는 반대로 반드시 HTTP 메시지의 Body 부분에 추가하고자 하는 값을 첨부하여 서버단에 있는 저장소에 해당 데이터를 추가하기 위한 목적으로 사용되는 메소드입니다.

### HTTP 와 HTTPS 차이점 (중요)
* HTTP와 HTTPS의 가장 큰 차이점은 HTTP 요청 메시지에 암호화를 적용 했는지 안했는지 여부입니다.
* HTTP는 클라이언트와 서버가 데이터를 주고 받기 위한 프로토콜 중 하나입니다. HTTP는 서버에 암호화 되지 않은 요청 메시지를 서버로 전달합니다. 이런 경우 데이터가 날것 그대로 전달되기 때문에 중간에 요청 메시지가 유출 혹은 변조될 위험이 존재합니다.
* HTTPS는 HTTP에 암호화가 추가된 프로토콜로 데이터를 암호화하여 만약 데이터가 중간에 유출되어도 복호화키가 없다면 어떤 데이터인지 알 수 없도록한 방식입니다. 
* HTTPS는 암호화를 위해 인증서와 한쌍의 복호화 키를 가지고 진행됩니다. 서버가 클라이언트로부터 요청을 받는다면 인증서, 응답 객체, 공개키를 클라이언트에게 보내게 됩니다. 응답 받은 클라이언트에서는 인증서에 적히는 도메인이 자신이 보낸 도메인과 같은지를 비교하여 데이터가 중간에 다른 도메인을 거쳐서 자신에게 도착했는지 확인할 수 있게 됩니다.
* HTTPS는 공개키와 비밀키 개념을 기반으로 하는 비대칭 키 암호화 방식을 사용합니다. 클라이언트와 서버 각각 한쌍의 키를 가지고 있고, 만약 클라이언트가 서버에게 메시지를 보내고 싶은 경우 hand shake단계에서 각자의 공개키를 서로에게 전달 받게 됩니다. 그 후, 전달 받은 공개키로 데이터를 암호화하여 http메시지를 서로에게 보내게 됩니다. 서버는 자신의 공개키로 암호화된 클라이언트의 메시지를 개인키로 복호화하여 전달 받은 메시지를 읽을 수 있게 되는 단계로 진행 됩니다.

### http status code 의 200대 300대 400대 500대 마다 각각 어떤목적을 가진 응답코드인가요.
* 200대의 상태코드는 주로 응답 성공을 의미하고,
* 300대의 상태코드는 주로 요청을 완료하기 위해 추가적인 작업이 필요한 경우
* 400대의 상태코드는 주로 클라이언트의 요청 에러의 의미를 갖고
* 500대의 상태코드는 주로 서버의 에러 의미를 가지고 있습니다.

### OOP가 무엇인가요.
* 객체 지향 프로그래밍으로 불리며, 객체에 역할과 책임을 추상화시켜 서로 협력하여 하나의 기능을 완성시키는 프로그래밍 패러다임 중 하나로서 캡슐화, 추상화, 상속, 다형성이라는 주요 개념이 존재합니다.
### OOP의 장점과 단점 ?
* 장점으로는 객체를 각각의 역할과 책임에 따라 생성하여 조립하는 방식으로 기능을 구현하기 때문에, 반복되는 역할을 재사용하기 쉽고, 하나의 로직을 여러곳에서 재사용 하기 때문에 한곳에서만 수정하면 모두 수정되어 유지보수 측면에서도 유리하게 됩니다. 또한, 클래스 단위로 모듈화하기 때문에 업무를 분담해서 개발하는 것이 편합니다.
* 단점으로는 우선 모든 역할과 책임을 적절하게 나눠서 로직을 구성해야하기 때문에 설계하는 과정이 어렵고 시간이 오래걸립니다. 또한, 사용하는 객체가 너무 많아지는 경우, 즉 역할을 너무 세분화하여 만드는 경우에는 용량이 커지는 단점이 있습니다.

### 캡슐화가 어떤건지 설명해주세요.
* 변수나 메서드를 가진 인스턴스를 생성하기 위한 클래스를 정의하여 외부에서는 로직을 볼 수 없도록 하는 것을 캡슐화라고 합니다.

### 추상화는 어떤 개념인가요?
* 내부의 복잡한 로직의 핵심적인 기능을 하나의 모듈로 만들어서 핵심만 표현하는 것을 추상화라고 합니다.

### 상속에 대해서 설명해주세요
* 상속은 하위 클래스에서 상위 클래스의 변수와 메서드에 접근 할 수 있게 하기 위한 개념으로 클래스간 관계를 형성하기 위한 개념입니다. 이미 만든 로직을 다시 만들지 않고 재사용 하기 위해 사용하게 됩니다.
* 하지만, 상속의 단점으로는 상위 클래스와 강한 결합해 있기 때문에 응집력이 떨어지게 됩니다. 이는 상위 클래스에서 메서드의 로직을 변경하게 된다면, 하위 클래스에서 모두 바꿔줘야 하는일이 발생하게 됩니다.
* 상속은 클래스의 역할을 확장하는데 사용하는 것이 아니라, 수정의 용도로 사용해야 합니다.
### 다형성이라는 것은 어떤 것인가요? 
* 다형성이라는 것은 변수명과 메서드명이 같아도 다른 로직을 불러 올 수 있다는 개념으로 오버라이딩과 오버로딩의 개념과 이어지게 됩니다.
* 다형성은 변화에 유연한 코드를 만들때 반복되는 코드 작성을 줄이며, 꼭 필요한 코드를 수정한다는 장점이 있습니다.
### 오버로딩, 오버라이딩이란? 
* 오버라이딩이란 부모 클래스의 변수나, 메서드를 자식 클래스에서 재정의하여 사용하는 것을 오버라이딩이라고 하며,
* 오버로딩이란 같은 이름의 함수를 여러개 정의하여, 매개변수의 타입이나, 개수를 다르게하여 호출하는 방법을 오버 로딩이라고 합니다. 

### 클래스와 인스턴스는 무엇인가요? 
* 클래스는 어떤 기능을 완성하기 위한 로직들을 모아 변수나 메서드를 정의한 하나의 설계도입니다.
* 인스턴스는 클래스에서 정의한 것들을 토대로 실제로 만들어진 객체를 인스턴스라고 합니다.

### 조인이 뭔가요. Inner join과 outer join의 차이는 무엇인가요.
* inner join은 두개의 테이블에서 조건절을 만족하는 교집합 부분을 가져오고자 하는 경우 사용하게 됩니다. 하지만, outer join 같은 경우 left와, right로 나뉘며 두 개의 테이블에서 조건절을 만족하는 부분과 더해서 left/right 테이블의 모든 속성을 가져오고 싶은 경우 사용하게 됩니다.

### REST API가 무엇인지 설명해주세요.
* REST는 Representational State Transfer의 약자이고 API는 Application Programming Interface의 약자입니다. 우선 API는 사람이 가게에 가서 음료수를 주문하는 경우에 보게 되는 매뉴판에 비교할 수 있으며, 사람이 아닌 프로그램 간 주문하기 위한 매뉴판을 API라고 합니다.
* 앞에 REST라는 단어가 붙은 이유는 프로그램 간 사용하는 매뉴판이지만 결국 사람이 작성하고 읽어야 하기 때문에 서로간에 작성 규칙을 적용하여 누가 읽어도 이해할 수 있도록 하기 위해서 입니다. 따라서, RESTful 하다는 말은 이 규칙이 잘 적용된 REST API를 뜻합니다.

### cors에서 origin이 뭔가요.
* cors의 origin은 요청을 허가하는 클라이언트의 URL입니다.

* CORS란 무엇이고 목적은 어떻게 되는지 설명해주세요. 그리고 NodeJS 기반의 웹 서버에서 CORS를 어떻게 설정하는지 설명해주세요.
CORS (Cross-Origin-Resource-Sharing)의 약자로 한글로 해석하자면 교차 출처 자원 공유로 할 수 있습니다. CORS는 기존의 웹 브라우저의 설정과는 반대로 클라이언트의 URL과 서버의 URL이 달라도 클라이언트의 요청을 서버에서 허가할 수 있는 설정입니다. 

* 기존의 웹 브라우저는 사용자가 브라우저 주소창에 URL을 작성하여 요청을 서버에 넣는다면 Server에서 비즈니스 로직과 더불어 HTML 문서까지 반환하는 방식인 SOP(Same Origin Policy)인 동일 출처 정책을 적용하는 것이 가장 일반적인 방법이었습니다. 이에 따라 웹 브라우저에서 만약 다른 도메인에서 요청이 들어오는 것은 보안상 위험한 문제로 인식하는 시기 였기 때문에 기본적으로 웹 브라우저에서 해당 요청을 자체를 막는 설정이 적용되어 있었습니다. 하지만, 웹사이트가 발전 할수록 서버에서 단순하게 HTML 문서를 반환하는 일 뿐만 아니라 사용자와의 인터렉션에 대한 요구가 크게 증가하게 되었습니다. 이러한 요구는 다양한 데이터를 사용자에게 제공하기 위해 기존에 같은 출처에만 요청을 보내는 것보다 다른 출처에 방대한 데이터를 요구하는 수요가 크게 증가하게 되었습니다. 이러한 요청 공식적으로 셋팅하는 방법이 CORS 설정이라고 알고 있습니다.  
### Node.js 기반의 웹서버에서 CORS를 설정하는 방법은 ? 
제가 구축했던 express를 기반 서버에서 cors 설정을 위해 사용한 방법은 매우 간단해습니다. express라는 라이브러리에서 use라는 메서드로 cors 미들웨어 함수를 실행하였습니다. 매계변수 자리에 origin이나, 요청 메서드 등의 설정을 적용하면 손쉽게 cors가 적용되었습니다. 특히 origin부분에 출처인 URL을 적용해야 하는데 주의해야 했던 부분은 프로토콜 & 호스트 & 포트를 요청하는 URL과 일치시켜야 요청을 허가했습니다.
